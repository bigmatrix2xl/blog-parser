# Как передать решение клиенту

Есть 2 рабочих сценария.

## Вариант A. Ты делаешь все сам

Подходит, если:
- нужно быстро получить результат
- клиент не хочет разбираться в Weavy
- важно контролировать качество руками

Схема:
1. Локально запускаешь парсер
2. Получаешь папки со скачанными картинками и manifest
3. По одной статье прогоняешь пачки через Weavy App
4. Отдаешь клиенту готовые новые картинки + таблицу учета

Плюсы:
- меньше шансов, что клиент сломает workflow
- качество выше
- проще держать единый стиль

Минусы:
- ты становишься узким местом
- все операционные действия на тебе

---

## Вариант B. Ты собираешь систему, клиент пользуется сам

Подходит, если:
- картинок очень много
- клиент готов работать внутри Weavy
- хочется передать рутинную часть

Схема:
1. Ты создаешь один production workflow
2. Публикуешь его как `Design App`
3. Делаешь для клиента короткую инструкцию
4. Клиент сам загружает картинки по одной статье и получает результаты

Плюсы:
- масштабируется
- клиенту не нужно видеть внутренние ноды
- можно использовать как полу-готовый сервис

Минусы:
- клиенту все равно потребуется минимальное обучение
- нужен контроль качества на его стороне

---

## Что я рекомендую

Лучший компромисс:

### Этап 1
Ты делаешь сам первые 3–5 партий статей.

Зачем:
- допилишь правила отбора
- поймешь реальные сбои
- увидишь, где клиент будет путаться

### Этап 2
После стабилизации публикуешь `Design App` и передаешь клиенту уже не ноды, а форму.

То есть клиенту показываешь не canvas, а:
- загрузку картинок
- title статьи
- context статьи
- кнопку Run

Это самый понятный вариант передачи.

---

## Что обязательно передать клиенту вместе с App

1. Простую инструкцию в 1 странице:
   - одна статья = один запуск
   - сначала title
   - потом 2–3 вводных абзаца
   - потом картинки этой статьи
   - нажать Run
   - скачать результат

2. Таблицу решений:
   - include
   - review
   - skip

3. Правило качества:
   - если картинка слишком похожа на исходную — не брать
   - если получился бренд / текст / логотип — не брать
   - если смысл статьи потерялся — перегонять заново

4. Кто отвечает за финальный апрув
   - без этого начнется хаос

## Организация папок

```text

  gid-po-vyboru-kondicionera-dlya-doma/
  kak-spilili-glavnuyu-el-strany/

02_generated_candidates/
  gid-po-vyboru-kondicionera-dlya-doma/
  kak-spilili-glavnuyu-el-strany/

  gid-po-vyboru-kondicionera-dlya-doma/
  kak-spilili-glavnuyu-el-strany/
```

И тот же `article_slug` использовать в manifest.

## Кто должен владеть workflow в Weavy

Лучше всего:
- workflow создаешь ты
- публикуешь ты
- клиенту даешь доступ по ссылке / шарингу
- у тебя остается master-версия

Так безопаснее, чем если клиент будет редактировать единственный оригинал.

---

## Практическое решение

Я бы шел так:

1. Сначала запускаем локальный парсер и собираем весь массив картинок
2. Проверяем rules с клиентом
3. Настраиваем один master-flow в Weavy
4. Превращаем его в `Design App`
5. Первые партии прогоняешь ты
6. Потом, если процесс стабилен, передаешь клиенту как self-service

Это не только реально автоматизировать — это как раз тот случай, где automation дает наибольшую пользу.
